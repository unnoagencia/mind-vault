# Web Dashboard — Design

**Date:** 2026-04-13
**Status:** Draft

## Summary

Add a read-only web dashboard to Mind Vault so the owner can browse notes and explore their correlations visually. Protected by the same credentials already used for the MCP OAuth flow, but via a separate cookie session (OAuth is for MCP clients, not browsers).

Two views:

1. **Notes** — simple list, click to open note detail with markdown render and clickable outbound links.
2. **Graph** — Obsidian-style force-directed visualization, WebGL, performant for 1000+ notes.

## Goals

- Single-user read-only access to existing vault content
- Visual "wow" factor to showcase the latticework structure
- Fluid at 1000+ notes (60fps pan/zoom target)
- Zero impact on existing MCP OAuth flow

## Non-goals (MVP)

- Search, filtering, pagination
- Note editing/creation from web
- Light mode / theme toggle
- Multi-user support
- Mobile optimization (works, but not polished)

## Architecture

Same Worker, new module `src/web/`. The existing `authHandler` gets a top-level branch that routes `/app/*` before the OAuth flow touches anything.

### Routes

```
GET  /app              → redirect to /app/login (no cookie) or /app/notes (cookie valid)
GET  /app/login        → login form (Midnight Nebula style)
POST /app/login        → validate OWNER_EMAIL + OWNER_PASSWORD_HASH, set session cookie
POST /app/logout       → clear cookie, redirect to /app/login
GET  /app/notes        → server-rendered HTML list
GET  /app/notes/:id    → server-rendered HTML note detail
GET  /app/graph        → HTML shell + Sigma.js bundle
GET  /app/graph/data   → JSON { nodes, edges } with precomputed layout
```

OAuth endpoints (`/authorize`, `/token`, `/register`, `/mcp`) remain untouched.

### Session

Cookie `mv_session`:
- `HttpOnly`, `Secure`, `SameSite=Lax`, `Max-Age=604800` (7 days), `Path=/app`
- Value format: `base64url(email).base64url(issuedAt).base64url(HMAC-SHA256(email + "|" + issuedAt, SESSION_SECRET))`
- Verification: recompute HMAC, constant-time compare, check `issuedAt + 7d > now`
- Middleware `requireSession(req, env)` runs on all `/app/*` except `/app/login`; redirects to `/app/login?next=<path>` on failure

### Session secret

New Worker secret `SESSION_SECRET` (32-byte random, hex). Generated by the setup wizard alongside email/hash, surfaced in `/setup/credentials` with a `wrangler secret put` command.

### CSRF

Login POST verifies `Origin` header matches `url.origin`. No cross-origin POSTs expected.

### Rate limit

Login endpoint: 5 failed attempts per IP per minute, stored in KV (reuse `GRAPH_CACHE` namespace or a new `RATE_LIMIT` binding — decide in plan).

## Views

### Login (`/app/login`)

Same visual as existing `/authorize` login form — email + passphrase, Midnight Nebula dark theme. On success sets cookie and redirects to `next` param or `/app/notes`.

### Notes list (`/app/notes`)

Layout:
- **Sidebar (left, ~180px)**: logo `◆ Mind Vault`, nav items `Notes` (active) / `Graph`, bottom: user email + `Log out`
- **Main**: header `Notes · N`, then list of cards

Each card: title, domain badges, `updated_at` date. Ordered by `updated_at DESC`. Whole card is a link to `/app/notes/:id`. No pagination in MVP.

### Note detail (`/app/notes/:id`)

Same shell. Main content:
- Title (h1)
- Domain badges
- `Created` / `Updated` metadata line
- Markdown-rendered body (use existing markdown renderer or add `marked` / `micromark` — decide in plan based on bundle size)
- `Connected to:` section listing explicit outbound links (from the `links` table). Each item: `→ <other note title>` as an `<a href="/app/notes/:other_id">` anchor, with the `why` and `relation_type` shown as subtitle.

404 if `id` not found.

### Graph (`/app/graph`)

Same shell. Main content = full-bleed WebGL canvas (Sigma.js).

Overlay in top-left corner (subtle, semi-transparent):
- `N notes · M links`
- Legend: solid line = explicit link, dashed line = similarity

Node click → `window.location = /app/notes/:id`.
Node hover → highlight node + neighbors, fade rest to 15% opacity.
Pan/zoom via mouse wheel + drag, standard Sigma defaults.

## Graph data pipeline

### Endpoint `/app/graph/data`

Returns:

```json
{
  "nodes": [
    { "id": "note_id", "label": "title", "domain": "infra", "size": 4, "x": 123.4, "y": -56.7 }
  ],
  "edges": [
    { "id": "e1", "source": "a", "target": "b", "type": "explicit", "why": "..." },
    { "id": "e2", "source": "a", "target": "c", "type": "similar", "score": 0.82 }
  ]
}
```

### Nodes

- One per note
- `size = 1 + log(degree + 1)` — degree counts explicit + similarity edges on the node
- `domain` = first entry in the note's `domains` array (notes can have multiple; we pick the first for coloring). Color determined client-side via deterministic hash → HSL hue in a Midnight Nebula-compatible palette (lavender, cyan, pink, violet range)
- `x`, `y` = precomputed layout coordinates (see below)

### Edges

- **Explicit**: `SELECT source_id, target_id, why, relation_type FROM links`
- **Similarity**: for each note, query Vectorize for top-3 nearest neighbors with `score > 0.75`. Dedupe symmetric pairs. Skip pairs that already have an explicit edge (explicit wins).

### Layout

Precomputed on the server using `graphology-layout-forceatlas2`:
- Build `graphology` graph from nodes + edges
- Run `forceAtlas2` for 500 iterations with `scalingRatio: 10, gravity: 1, barnesHutOptimize: true`
- Write `x`, `y` back onto each node
- Client renders fixed positions, no physics loop — this is the key to 1000+ node fluidity

### Cache

KV namespace `GRAPH_CACHE`, single key `graph:v1`.

Cache entry:
```json
{
  "version": 1,
  "computedAt": 1712990000,
  "sourceHash": "<max updated_at from notes + links>",
  "payload": { "nodes": [...], "edges": [...] }
}
```

Invalidation logic on `/app/graph/data`:
1. Query `SELECT MAX(updated_at) FROM notes`, same for `links`
2. Hash = `max(notes.updated_at, links.updated_at)`
3. Read cache. If cache hit AND `sourceHash` matches → serve
4. Otherwise recompute, write back, serve

First request after any change pays the recompute cost (~1-3s for 1000 notes). Subsequent requests hit cache (~10ms KV read).

### Similarity freshness

Similarity edges are sensitive to Vectorize's eventual consistency (1-2 min). Cache invalidation by `updated_at` is enough — stale similarity for 1-2 min is acceptable for a browsing dashboard.

## Performance targets

- Initial `/app/graph/data` response: <100ms on cache hit, <5s on cache miss at 2000 nodes
- Client render: 60fps pan/zoom up to 2000 nodes on a mid-range laptop
- Bundle size: Sigma.js + graphology + app code < 250KB gzipped

## Security

- All `/app/*` responses (except `/app/login` GET) require valid session cookie
- `SESSION_SECRET` is a Worker secret, never logged
- `Origin` check on login POST
- Rate limit on login (5 attempts/IP/min)
- `Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'` on `/app/*`
- Note content is rendered from trusted owner input, but still pass through markdown-safe rendering (no raw HTML)

## Dependencies (new)

- `sigma` — WebGL graph renderer
- `graphology` — graph data structure
- `graphology-layout-forceatlas2` — server-side layout
- Markdown renderer — `marked` or `micromark` (pick smallest, decide in plan)

Worker bundle impact to be measured in plan. If any dependency pushes us over the 1MB Worker limit, fall back to a lighter choice.

## KV binding

Add to `wrangler.toml`:

```toml
[[kv_namespaces]]
binding = "GRAPH_CACHE"
id = "<to be filled by user during setup>"
```

Setup wizard `/setup/credentials` screen gets:
1. New `SESSION_SECRET` generated alongside the password hash, with its own `wrangler secret put` command
2. Instructions to create the KV namespace: `wrangler kv namespace create GRAPH_CACHE` and paste the returned ID into `wrangler.toml`
3. Then `wrangler deploy`

## Visual style — Midnight Nebula

Shared across all `/app/*` pages:
- Background: `radial-gradient(ellipse at 40% 30%, #1a1438 0%, #0a0618 70%)`
- Primary text: `#e8dcff`
- Secondary text: `rgba(232, 220, 255, 0.55)`
- Accent palette (nodes, badges, highlights): lavender `#b48cff`, cyan `#8cc8ff`, pink `#ff9ad5`, violet `#a78bfa`
- Borders: `rgba(180, 140, 255, 0.1)`
- Cards / panels: `rgba(255, 255, 255, 0.03)` with `1px` lavender-tinted border
- Node glow in the graph: `blur(10px)` halo at 50% opacity, same color as the node

## Testing

- `src/web/session.test.ts` — sign/verify round-trip, tampered cookie rejected, expired cookie rejected, missing secret error
- `src/web/auth.test.ts` — login with right credentials sets cookie, wrong credentials returns 401, missing cookie on `/app/notes` redirects to login with `next` param
- `src/web/notes.test.ts` — list renders all notes sorted by `updated_at`, note detail renders links as anchors, 404 on unknown id
- `src/web/graph.test.ts` — `/graph/data` returns correct shape, cache hit on second call, invalidation when `updated_at` changes, similarity edges deduped against explicit
- `src/web/layout.test.ts` — forceatlas2 runs on a fixture graph and returns finite x/y for every node
- All via Miniflare like existing tests

## File layout

```
src/web/
  handler.ts          — route dispatch for /app/*
  session.ts          — sign, verify, middleware
  login.ts            — GET/POST /app/login
  notes.ts            — GET /app/notes, GET /app/notes/:id
  graph.ts            — GET /app/graph (shell)
  graph-data.ts       — GET /app/graph/data (compute + cache)
  layout.ts           — forceatlas2 wrapper
  similarity.ts       — Vectorize top-k + dedupe
  render.ts           — shared shell HTML (sidebar, head, nebula CSS)
  markdown.ts         — safe markdown → HTML
  styles.ts           — Midnight Nebula CSS constants
  client/
    graph.ts          — Sigma.js initialization, node click, hover
    graph.bundle.js   — built output served as /app/graph/bundle.js
```

Keep files small and single-purpose. Tests live next to the module.

## Out of scope

Explicitly deferred — do not implement in this iteration:

- Search / filter / pagination
- Creating or editing notes from the dashboard
- Light theme
- Mobile-specific layouts
- Multi-user / sharing
- Export / import
- Graph filtering by domain
- Different layout algorithms / user-tunable physics
